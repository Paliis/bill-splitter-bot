# Архітектура продукту: зараз PWA, з задумом на магазини

Цей документ фіксує **цільову архітектуру**, якої варто дотримуватися при додаванні **веб-PWA** та подальшому винесенні в **App Store / Google Play**. Поточний репозиторій — насамперед **Telegram-бот**; API й PWA з’являться як окремі модулі/репозиторії або папки в монорепо — але **принципи нижче незмінні**.

---

## 1. Єдине джерело правди — бекенд

- Доменні правила (**активна поїздка на контекст**, витрати, поділ, баланси, **settlement** при завершенні) живуть **на сервері** й викликаються клієнтами через **стабільний версіонований API** (наприклад `/v1/`).
- Клієнти (**PWA**, майбутні **iOS/Android**, за бажанням бот) **не дублюють** розрахунок боргів і переказів; лише показують результат і збирають ввід.
- **OpenAPI** (або еквівалент) як контракт: одна схема → веб зараз, потім **codegen** під мобайл без ручного роз’їзду типів.

Практичний крок: винести наявну логіку з `services/`, `handlers/` у **викликаєший шар** (HTTP), щоб бот і HTTP використовували ті самі функції роботи з БД.

---

## 2. Користувачі та сесії (не тільки Telegram)

- У продуктовій моделі **первинний** користувач — запис у **власній БД** (`users`), а не `telegram_user_id` як єдиний ключ.
- **Telegram** — один із **провайдерів автентифікації / прив’язки** (як Google / Apple пізніше): таблиця або поля на кшталт `auth_providers(provider, external_id, user_id)`.
- **PWA (зараз)** — **JWT (access + refresh)** або **сесії в httpOnly cookies**; той самий механізм **скидається на натив** (refresh у secure storage).
- OAuth (Google, Apple) можна підключити пізніше без зміни доменних сутностей, якщо з **першого дня** не прив’язувати бізнес-логіку до полів Telegram.

---

## 3. Клієнт тонкий, парсинг «одного рядка» — на сервері

- Валідація форм і зручність UX — на клієнті; **парсинг** вводу на кшталт **`100 кава` / `Кава 100`** — **в одному місці на бекенді** (або спільна бібліотека, якщо колись винесете в пакет, але не розмножуйте логіку між PWA і ботом без потреби).
- Запити до API — **структуровані поля** опційно + поле `raw_line`, якщо потрібен єдиний UX “одним рядком”.

---

## 4. PWA з задумом на стори

- **HTTPS**, **Web App Manifest**, **Service Worker** (мінімум: кеш shell / статики; повноцінний офлайн-CRUD — за потреби пізніше).
- **Маршрутизація**: осмислені URL (`/trips/:id`, `/expenses/new`) — згодом ті самі шляхи можна відкривати через **Universal Links / App Links** з нативної обгортки або в Flutter/React Native.
- **Push**: у мобільному вебі обмежено (особливо iOS); у схемі БД/API закладіть **місце** під реєстрацію **device tokens** (FCM / APNs) — навіть **stub**, щоб не ламати міграції пізніше.

---

## 5. Рекомендована структура репозиторію (коли з’явиться веб)

Варіанти, узгоджені з цим документом:

- `backend/` (FastAPI тощо) — HTTP + той самий домен, що й бот; або
- монорепо: `apps/web` (PWA), `apps/bot` (існуючий воркер), `packages/api-client` (згенерований з OpenAPI).

Головне — **один контракт API**, а не кількість папок.

---

## 6. Анти-патерни (щоб не ускладнити вихід у стори)

- Єдиний канал даних для продукту лише через **Telegram webhook** без власного API.
- Бізнес-правила, що живуть лише у FSM бота та **не викликаються** з HTTP.
- Неможливість ідентифікувати користувача **поза Telegram** (немає `users` / токенів).
- Роз’їзд логіки поділу витрат між PWA і сервером.

---

## 7. Фази без втрати фокусу

1. **HTTP API + PWA MVP** — логін (або код на email), трип, витрати, статус, завершення; усе через API.
2. **PWA** — manifest, SW, базова якість “встановити на екран”.
3. **Магазини** — нативний клієнт або обгортка до того ж API; Sign in with Apple/Google; пуші за потреби.
