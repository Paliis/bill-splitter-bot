# Архітектура продукту: зараз PWA, з задумом на магазини

Цей документ фіксує **цільову архітектуру**, якої варто дотримуватися при додаванні **веб-PWA** та подальшому винесенні в **App Store / Google Play**. Поточний репозиторій — насамперед **Telegram-бот**; API й PWA з’являться як окремі модулі/репозиторії або папки в монорепо — але **принципи нижче незмінні**.

---

## 0. Узгоджені рішення (колишні «відкриті питання»)

Усе нижче — **зафіксована політика продукту**; зміни лише через явне рішення й оновлення цього розділу.

### 0.1 Контекст групи (party), не лише Telegram-чат

- **Party (простір події)** — абстракція: «група людей + обмеження “одна активна поїздка”».
- У **Telegram** `party` однозначно мапиться на `chat_id` (як зараз).
- У **PWA** той самий контракт: party створюється вебом (запрошення, лінк); **одна активна поїздка на party**, як і для чату в боті.
- **Single source of truth**: дані поїздки й витрат прив’язані до **trip_id** і **party_id** у БД, а не до каналу UX.

### 0.2 Життєвий цикл поїздки / події

- Статуси: **`active`** → **`settled`** (підсумок і перекази зафіксовані в продукті) → **`archived`** (опційно, історія без редагування витрат або з обмеженим редагуванням — уточнювати в API при імплементації).
- Правило **один `active` на party** без змін.

### 0.3 Settlement (розрахункове ядро)

- З **MVP API** одразу виводимо **зведені мінімальні перекази** (той самий клас алгоритмів, що й у боті зараз), а не попарний повний граф «кожен з кожним».
- **Нетто-баланси** по учасниках — завжди узгоджені з тим, що показуємо в UI; копійкове вирівнювання — на бекенді.

### 0.4 Валюта

- **Окрема базова валюта на кожну поїздку / подію**, **обирається при створенні** (список підтримуваних кодів узгоджений у API).
- **Немає** жорсткого «тільки UAH для України»: українська **мова** не прив’язана до гривні; можна вести поїздку в EUR тощо.

### 0.5 Мова інтерфейсу (locale)

- **Збережена уподобання на користувача** у профілі (`users.locale` або еквівалент): `uk` / `en` (розширення — за потреби).
- **Перевизначення на сесію** (PWA): cookie, query або explicit toggle в UI — має пріоритет над профілем **лише для цієї сесії** (і опційно «запам’ятати в профіль»).
- **Telegram-бот**: до появи профілю з API — **fallback** як зараз (`language_code`); після зв’язування з `user_id` — **пріоритет** профіль + команда/кнопка «мова» за бажанням.
- Усі тексти клієнтів залишаються ключами i18n; **мова не змішується** з валютою поїздки.

### 0.6 Один людей — один `user_id` (зв’язування акаунтів)

- Перехід **Telegram ↔ PWA** вимагає **явного лінкування** (magic-link з бота, deep-link із токеном, Telegram Login Widget на вебі тощо) — щоб не плодити дублікати.
- Політика: **кілька провайдерів** на одного `user_id`; злиття дублікатів — окремий адмін/саппорт-флоу або керований merge пізніше.

### 0.7 Агрегований баланс / dashboard (кілька поїздок)

- **Заборонено** показувати **одну суму** «ваш загальний баланс» через додавання різних валют.
- Дозволено: **окремий блок на активну поїздку**, або **групування по валюті** (підсумок лише в межах однієї валюти), або список поїздок із балансом у **її** валюті.

### 0.8 «Confirm payment» у UI settlement

- Семантика: **не платіжний шлюз**, а **позначка виконання / взаємного підтвердження** (наприклад «я відправив» / «ми звірились»); стан зберігається в БД за потреби для UX, **гроші не рухає**.

### 0.9 Редагування учасників витрати (drag-and-drop)

- **MVP PWA**: той самий патерн, що й у боті — **перемикачі / список з галочками** (і «на всіх»).
- **Drag-and-drop** — після MVP, якщо буде запит (складніше в accessibility й тестах).

### 0.10 Продуктові рішення з роудмапу (RD-001…006)

Узгоджено з власником продукту; деталь — [ROADMAP.md](./ROADMAP.md).

- **Витрати:** редагувати / видаляти може **лише автор**; **історії змін (аудит)** у MVP **немає**.
- **Після settle:** подія в **архіві**, дані **без змін** після закриття (ціль для API); **перегляд** архіву минулих подій — так.
- **Вкладення (чеки):** у MVP **не** збираємо.
- **Монетизація:** зараз **донати**; **freemium** і **платний додаток у сторах** — можливо надалі (юридично зафіксовано в чернетці TERMS).
- **Канал пріоритету:** **PWA + магазини**; Telegram як вхід і WebApp — див. ROADMAP Фаза 3–4.
- **Юридичне:** чернетки [legal/](./legal/) — перед публікацією **юрист**.

## 1. Єдине джерело правди — бекенд

- Доменні правила (**активна поїздка на контекст**, витрати, поділ, баланси, **settlement** при завершенні) живуть **на сервері** й викликаються клієнтами через **стабільний версіонований API** (наприклад `/v1/`).
- Клієнти (**PWA**, майбутні **iOS/Android**, за бажанням бот) **не дублюють** розрахунок боргів і переказів; лише показують результат і збирають ввід.
- **OpenAPI** як контракт: [openapi/openapi.yaml](./openapi/openapi.yaml) — одна схема → веб зараз, потім **codegen** під мобайл без ручного роз’їзду типів.

Практичний крок: винести наявну логіку з `services/`, `handlers/` у **викликаєший шар** (HTTP), щоб бот і HTTP використовували ті самі функції роботи з БД.

---

## 2. Користувачі та сесії (не тільки Telegram)

- У продуктовій моделі **первинний** користувач — запис у **власній БД** (`users`), а не `telegram_user_id` як єдиний ключ.
- **Telegram** — один із **провайдерів автентифікації / прив’язки** (як Google / Apple пізніше): таблиця або поля на кшталт `auth_providers(provider, external_id, user_id)`.
- **PWA (зараз)** — **JWT (access + refresh)** або **сесії в httpOnly cookies**; той самий механізм **скидається на натив** (refresh у secure storage).
- OAuth (Google, Apple) можна підключити пізніше без зміни доменних сутностей, якщо з **першого дня** не прив’язувати бізнес-логіку до полів Telegram.

---

## 3. Клієнт тонкий, парсинг «одного рядка» — на сервері

- Валідація форм і зручність UX — на клієнті; **парсинг** вводу на кшталт **`100 кава` / `Кава 100`** — **в одному місці на бекенді** (або спільна бібліотека, якщо колись винесете в пакет, але не розмножуйте логіку між PWA і ботом без потреби).
- Запити до API — **структуровані поля** опційно + поле `raw_line`, якщо потрібен єдиний UX “одним рядком”.

---

## 4. PWA з задумом на стори

- **HTTPS**, **Web App Manifest**, **Service Worker** (мінімум: кеш shell / статики; повноцінний офлайн-CRUD — за потреби пізніше).
- **Маршрутизація**: осмислені URL (`/trips/:id`, `/expenses/new`) — згодом ті самі шляхи можна відкривати через **Universal Links / App Links** з нативної обгортки або в Flutter/React Native.
- **Push**: у мобільному вебі обмежено (особливо iOS); у схемі БД/API закладіть **місце** під реєстрацію **device tokens** (FCM / APNs) — навіть **stub**, щоб не ламати міграції пізніше.

---

## 5. Рекомендована структура репозиторію (коли з’явиться веб)

Варіанти, узгоджені з цим документом:

- `backend/` (FastAPI тощо) — HTTP + той самий домен, що й бот; або
- монорепо: `apps/web` (PWA), `apps/bot` (існуючий воркер), `packages/api-client` (згенерований з OpenAPI).

Головне — **один контракт API**, а не кількість папок.

---

## 6. Анти-патерни (щоб не ускладнити вихід у стори)

- Єдиний канал даних для продукту лише через **Telegram webhook** без власного API.
- Бізнес-правила, що живуть лише у FSM бота та **не викликаються** з HTTP.
- Неможливість ідентифікувати користувача **поза Telegram** (немає `users` / токенів).
- Роз’їзд логіки поділу витрат між PWA і сервером.
