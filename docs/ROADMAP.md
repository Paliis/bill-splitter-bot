# Роудмап і порядок робіт

Документ описує **етапи**, **артефакти**, які оновлювати після кожного кроку, і **відкриті питання**, що потребують вашого рішення **або додаткових даних**. Терміни й оцінки в **людино-днях не проставлені** — їх можна додати, коли будуть вхідні дані.

**Узгоджені архітектурні рішення** (валюта на поїздку, locale, party, settlement тощо): [ARCHITECTURE.md](./ARCHITECTURE.md) §0.

---

## Як рухаємось (домовленість)

1. **Покроково:** завершуємо крок → оновлюємо цей файл (статус етапу, закриті питання) + при змінах продукту/коду — [CHANGELOG.md](./CHANGELOG.md); при зміні правил продукту для людей — [PRODUCT.md](./PRODUCT.md).
2. **Невизначеність:** якщо для кроку **не вистачає даних** або є **продуктовий вибір** — **питаємо вас**, не заповнюємо припущеннями.
3. **Поточний фокус:** див. розділ **«Активний етап»** нижче — він змінюється лише явним рішенням після закриття попереднього кроку або зміни пріоритету.

---

## Артефакти (що і коли чіпати)

| Артефакт | Коли оновлювати |
|----------|------------------|
| **ROADMAP.md** (цей файл) | Початок/кінець кроку; нове відкрите питання; зміна активного етапу; закриття питання (позначити **Вирішено** + коротка формулювання рішення або посилання на § у ARCHITECTURE/PRODUCT). |
| **ARCHITECTURE.md** | Нова домовленість по системі (API, дані, безпека); зміни в §0 за потреби через вас. |
| **PRODUCT.md** | Зміна сценаріїв або правил для користувача, нові команди/обмеження. |
| **CHANGELOG.md** | Будь-яка змістовна зміна в репозиторії (код, доки, деплой-конфіг). |

| Поле | Значення |

| **Наступна дія** | Вирішити **RD-001 … RD-006** (хоча б на рівні MVP) або явно відкласти кожен пункт з міткою «після MVP». |

## Фази (огляд)

### Фаза 0 — Підготовка і вирівнювання

- [x] ARCHITECTURE §0 (узгоджені рішення).
- [x] ROADMAP + перелік відкритих питань.
- [ ] Рішення по **RD-001 … RD-006** (або явний defer).
- [ ] Скелет **OpenAPI** або список ендпоінтів v1 (без реалізації — за потреби після RD).

### Фаза 1 — Ядро: HTTP API + спільний домен з ботом

- [ ] Виділити/імплементувати **REST API v1** (party/trip, members, expenses, balances, settle) на існуючій БД або з міграціями.
- [ ] **Settlement** — той самий алгоритм, що й у боті (мінімізація переказів); покрити тестами.
- [ ] **Бот** або викликає API, або спільний сервісний шар без дублювання правил (узгодити на кроці імплементації після RD).
- [ ] **Авторизація API**: заготовка під JWT та мапінг Telegram → `user_id` (мінімум для внутрішніх викликів).

### Фаза 2 — PWA MVP

- [ ] Проєкт PWA (окрема папка/репо — рішення на Фазі 1).
- [ ] Логін (пріоритет: **Telegram** для існуючих користувачів за [ARCHITECTURE.md](./ARCHITECTURE.md) §0.6).
- [ ] Екрани: список поїздок / деталь, додати витрату, статус, settlement (відображення результату API).
- [ ] **Manifest + Service Worker** (базово).
- [ ] **Мова:** профіль + сесійне перевизначення (як у §0.5).
- [ ] **Валюта:** вибір при створенні поїздки (§0.4).

### Фаза 3 — Інтеграція з Telegram як каналом до PWA

- [ ] **WebApp / Mini App** (кнопка відкриття PWA з чату) — за наявності домену та налаштувань BotFather.
- [ ] Deep links на конкретну витрату/поїздку — після стабільних `id` у API.

### Фаза 4 — Глобальне зростання і магазини (після PMF за вашим рішенням)

- [ ] OAuth Google / Apple, пуші (заготовка під токени вже в архітектурі).
- [ ] Нативні обгортки або крос-платформений клієнт до того ж API.

*Склад і порядок Фази 3/4 можна переставити після ваших пріоритетів — зміни фіксуються тут.*

## Відкриті питання для вирішення

Статуси: **Відкрито** | **Відкладено** | **Вирішено** (дописати рішення в колонці).

| ID | Питання | Статус | Рішення / коментар |
|----|---------|--------|---------------------|
| **RD-001** | Хто може **редагувати / видаляти** витрату після створення (автор, адміни групи, всі учасники)? Чи потрібна **історія змін** в MVP? | Відкрито | |
| **RD-002** | Після **settled**: лише **архів / read-only**, чи дозволено **коригування / перевідкриття** поїздки? | Відкрито | |
| **RD-003** | **Монетизація:** лише донати (Mono / BMAC / Donatello) чи планується **freemium** (обмеження учасників/поїздок) — хоча б принципово для текстів на сайті? | Відкрито | |
| **RD-004** | **Фото чеків / вкладення** до витрат у MVP API/PWA — так чи ні? (наслідки: сховище, приватність, вартість). | Відкрито | |
| **RD-005** | **Головний «дім» продукту** на 12–18 місяців: пріоритет **Telegram**, **PWA** чи **паралельно рівно** (впливає на порядок фіч між фазами)? | Відкрито | |
| **RD-006** | **Юридичний мінімум** до збору email/Google/Apple: чи готуєте **Privacy / Terms** самостійно чи потрібна лише технічна заготовка (експорт/видалення)? | Відкрито | |

*Нові питання додавати з наступним вільним ID (RD-007 …).*
